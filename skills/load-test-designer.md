# Load Test Designer

Go from "we should probably load test this" to a runnable k6 or Artillery script with realistic user-behaviour distributions, per-endpoint SLOs, and a monitoring checklist — in one pass.

## When to Use

- Preparing a new service or feature for production traffic
- Validating a performance fix under realistic concurrent load
- Setting a capacity baseline before a scaling event or infrastructure change
- Designing a spike test to find the breaking point of a system

## Process

1. **Collect the target endpoint list** — read the OpenAPI spec (if present) or a manually provided list:

   ```bash
   ls openapi.{yaml,yml,json} swagger.{yaml,yml,json} 2>/dev/null
   ```

   Extract: method, path, required parameters, auth scheme, expected response codes. For each endpoint, estimate the relative call frequency based on typical user flows (e.g. GET /products is called 10× more than POST /orders).

2. **Identify the load model** — classify the test scenario type:

   | Scenario | Pattern | Use when |
   |----------|---------|----------|
   | Steady-state | Constant VUs for 10 min | Measuring sustained throughput |
   | Ramp-up | 0 → peak over 5 min, hold 10 min, ramp-down | Validating autoscaling |
   | Spike | Instant 10× traffic for 1 min | Testing queue/buffer resilience |
   | Soak | Low constant VUs for 60+ min | Finding memory leaks and connection pool exhaustion |

   Default to a combined ramp-up + steady-state + spike if no specific scenario is requested.

3. **Define per-endpoint SLOs** — derive from the API type:

   | Endpoint type | p95 SLO | Error-rate budget |
   |---------------|---------|-------------------|
   | Read (GET) | ≤ 200 ms | < 0.1% |
   | Write (POST/PUT/PATCH) | ≤ 500 ms | < 0.5% |
   | Heavy compute / search | ≤ 2 000 ms | < 1.0% |
   | Health / readiness | ≤ 50 ms | 0% |

4. **Choose the tool** — prefer k6 for JavaScript environments; prefer Artillery for teams already using it; fall back to k6 if unspecified (wider CI integration, structured HTML reports).

5. **Generate the script** — produce a complete, executable file:

   **k6 template:**

   ```javascript
   import http from 'k6/http'
   import { check, sleep } from 'k6'
   import { Rate, Trend } from 'k6/metrics'

   const errorRate = new Rate('errors')
   const productListP95 = new Trend('product_list_p95', true)

   export const options = {
     stages: [
       { duration: '2m', target: 20 },   // ramp-up
       { duration: '5m', target: 20 },   // steady
       { duration: '1m', target: 100 },  // spike
       { duration: '2m', target: 0 },    // ramp-down
     ],
     thresholds: {
       'http_req_duration{endpoint:GET /products}': ['p(95)<200'],
       'http_req_duration{endpoint:POST /orders}':  ['p(95)<500'],
       errors: ['rate<0.01'],
     },
   }

   export default function () {
     // --- GET /products (weight: 60%) ---
     if (Math.random() < 0.6) {
       const res = http.get(`${__ENV.BASE_URL}/products`, {
         tags: { endpoint: 'GET /products' },
       })
       check(res, { 'products 200': r => r.status === 200 })
       errorRate.add(res.status !== 200)
       productListP95.add(res.timings.duration)
     }
     sleep(0.5 + Math.random() * 1.0)  // think-time 0.5–1.5 s
   }
   ```

6. **Write the monitoring checklist** — what to watch during the test run:

   - [ ] Request rate matches configured VUs (`k6 dashboard` or Grafana)
   - [ ] Error rate stays below budget per threshold
   - [ ] Database connection pool not exhausted (watch `pool_wait_duration` or `pg_stat_activity`)
   - [ ] Memory stable over soak duration (no sawtooth leak)
   - [ ] CPU on app servers stays below 70% during steady state
   - [ ] Downstream services not rate-limited (watch 429 count)

## Output Format

```javascript
// Generated load test — k6
// Target: api.example.com/v1
// Scenario: ramp-up + steady + spike
// Generated by Lilara load-test-designer on 2026-05-26

import http from 'k6/http'
import { check, sleep } from 'k6'
import { Rate } from 'k6/metrics'

const errors = new Rate('errors')
const BASE_URL = __ENV.BASE_URL || 'http://localhost:3000'

export const options = {
  stages: [
    { duration: '2m', target: 20  },
    { duration: '5m', target: 20  },
    { duration: '1m', target: 100 },
    { duration: '2m', target: 0   },
  ],
  thresholds: {
    'http_req_duration{endpoint:GET /v1/products}': ['p(95)<200'],
    'http_req_duration{endpoint:POST /v1/orders}':  ['p(95)<500'],
    errors:                                          ['rate<0.005'],
  },
}

export default function () {
  const roll = Math.random()

  if (roll < 0.55) {
    // GET /v1/products — 55% of traffic
    const r = http.get(`${BASE_URL}/v1/products`, {
      headers: { Authorization: `Bearer ${__ENV.API_TOKEN}` },
      tags: { endpoint: 'GET /v1/products' },
    })
    check(r, { 'status 200': res => res.status === 200 })
    errors.add(r.status !== 200)
  } else if (roll < 0.80) {
    // GET /v1/products/{id} — 25% of traffic
    const id = Math.floor(Math.random() * 1000) + 1
    const r = http.get(`${BASE_URL}/v1/products/${id}`, {
      headers: { Authorization: `Bearer ${__ENV.API_TOKEN}` },
      tags: { endpoint: 'GET /v1/products/{id}' },
    })
    check(r, { 'status 200 or 404': res => [200, 404].includes(res.status) })
  }
  // … remaining endpoints …

  sleep(0.5 + Math.random() * 1.5)
}
```

## Constraints

- Does not execute the load test — generates the script for review and human-triggered execution.
- Authentication credentials must be supplied at runtime via environment variables (`__ENV.API_TOKEN`, `__ENV.BASE_URL`); the generated script uses placeholders only.
- Traffic distribution weights are estimates based on common API usage patterns; tune them from real access logs or analytics before using in production capacity planning.
- Spike tests can overwhelm databases or downstream APIs that have no circuit breaker; coordinate with dependent-service owners before running a spike.
- k6 requires Node-compatible JavaScript (no `import` from npm in k6 scripts); all k6 imports use the `k6/` prefix from the k6 runtime, not npm.
